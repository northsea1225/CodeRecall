# CodeRecall 三方分工方案

> **原则**：Codex 主开发落地 · Gemini 主 UX/文档/创意 · Claude 低频做关键审阅（省额度）
> **一致度**：Codex 与 Gemini 自报分工基本完全一致，无冲突。

---

## 一、角色定位（一句话版）

| 角色 | Codex | Gemini | Claude |
|---|---|---|---|
| 头衔 | **主力工程师** | **产品/前端/文档** | **首席架构 + 质检** |
| 频次 | 高频（天级） | 高频（天级） | **极低频（周级）** |
| 核心价值 | 把方案写成可运行代码 | 把产品做得惊艳 | 关键时刻点睛 |

---

## 二、Codex 职责（主开发）

**擅长**：代码生成、算法实现、后端/重工程、测试、重构、脚手架

### W1 — 工程地基
- 前后端仓库结构、Vite/FastAPI 脚手架、开发环境打通
- SQLAlchemy 数据模型（mistakes/categories/tags/review_logs/sessions）+ Alembic 迁移
- 基础 CRUD API（mistakes）
- React 前端骨架、路由、API 封装、zustand store
- Monaco Editor 最小可用接入
- JSON 导入导出基础能力

### W2 — 核心业务
- 分类筛选 + 关键词搜索（`LIKE`）
- Review 流程后端：随机抽题 → 提交结果 → 写 review_logs
- **SM-2 算法实现** + 复习调度接口
- **AI 错因分析**前后端骨架：调 LLM API、结果存储、feature flag
- 首轮单元测试 + 接口测试（pytest）

### W3 — 增强与重构
- 统计接口与聚合逻辑（分类占比/错因 TOP/复习趋势）
- 导入导出增强：校验、容错、版本字段
- Monaco 体验增强：草稿保存、恢复、Code Diff
- 抽离复用 hooks / services / schemas
- 关键业务测试补齐

### W4 — 收口
- 全链路联调、Bug 修复、性能稳定性
- 边界 case 修复、数据一致性检查
- 打包发布、日志与错误处理
- 最终验收测试

---

## 三、Gemini 职责（产品/UX/创意）

**擅长**：视觉稿、交互设计、文档、README、Prompt 工程、演示脚本、多模态创意

### W1 — 设计先导
- UI 规范定义（配色/字体/组件风格，Linear/Notion 风）
- 关键页面 ASCII 原型图（Dashboard/Editor/Review/Stats）
- Vite+React 基础 UI 组件封装（Button/Card/Tag/EmptyState）
- README 1.0（项目口号、架构图占位、快速启动）

### W2 — 前端交互
- Monaco Editor 主题定制 + 代码高亮配置
- **AI 分析结果流式渲染** + 提示语优化
- 录题页/复习页交互细节打磨
- 空态/错误态/引导文案

### W3 — 可视化与内容
- ECharts/Recharts：复习进度热力图、错因饼图、错误分类柱图
- 前端 SM-2 复习引导 UI（Again/Hard/Good/Easy 按钮组）
- AI Prompt 精调（让错因分析更像导师而非翻译）
- README 2.0、docs/architecture.md、docs/demo-script.md

### W4 — 结项与演示
- **5 分钟演示视频脚本**（分镜 + 台词）
- 录制 Demo 操作流
- 答辩材料 / 项目结项报告
- 未来扩展方案（VS Code 插件、多模态拍照识别）

## 四、Claude 职责（极低频 · 省额度）

**擅长**：综合判断、跨文件精细编辑、代码 review、架构裁决
**原则**：不承担主开发，只做「低频 + 高价值 + 短时」任务

| 时机 | 任务 | 预估消耗 |
|---|---|---|
| W1 末 | **架构 sanity check**：扫一遍目录结构、API 契约、ORM 模型是否合理 | 1 次小对话 |
| W2 末 | **SM-2 + AI 集成定点 review**：算法正确性、边界、幂等、安全 | 1 次小对话 |
| W3 中 | **导入导出 + 统计聚合 review**：口径一致性、数据迁移安全 | 1 次小对话（可选） |
| W4 末 | **最终 PR 级审阅**：跨文件一致性、命名、回归风险、演示前盲点 | 1 次大对话 |
| 任何时候 | **Codex ↔ Gemini 冲突仲裁**：两方方案打架时做裁决 | 按需 |

**总计约 3–5 次对话**，尽量每次把要点打包问完，避免多轮反复。

## 六、协作规则

1. **Codex 是主提交者**：所有代码变更由 Codex 落地，Gemini 产出设计稿与前端组件建议后由 Codex 整合。
2. **Gemini 的交付物是「可直接贴进代码的 Markdown/JSX 片段 + 文档」**，不做大范围 repo 操作。
3. **Claude 只读 + 只评**：不主动写大段代码，只给精准建议和跨文件微调。
4. **冲突升级路径**：Codex ↔ Gemini 冲突 → 写成 1 条对比问题 → Claude 裁决。
5. **每周五对齐一次**：Codex 给进度，Gemini 给演示素材，Claude 做架构速扫（可跳过）。
