# CodeRecall AI Prompts v2 设计文档

## 1. Prompt v2 设计目标

### v1 的问题
- **输出不稳定**：AI 生成的格式不可控，有时包含冗长无意义的段落。
- **小节缺失**：分析结构不统一，经常会遗漏关键的复习指导点。
- **语言无差异**：未针对不同编程语言和题目类型提供特定的侧重点，导致分析泛泛而谈，缺乏针对性。

### v2 改进点
- **强约束标题**：严格要求 AI 输出固定的 4 个二级标题（Markdown 格式），确保前端解析和展示结构的一致性。
- **语言分支（Language Branch）**：根据用户输入代码的语言或题目类型，动态注入不同的 Prompt 侧重点，提供更精准的专家级分析。
- **越界防御**：明确告知 AI 角色为导师，只做根因分析和思路引导，**严禁直接提供完整的正确代码**，迫使学习者主动思考。

## 2. 固定 4 小节规范

为保证前端解析和展示的稳定性，Prompt v2 强制 AI 按照以下四个小节输出，每节均采用 `##` 作为标题格式。

| 小节标题 | 定义 | 期望字数 | 风格要求 |
| --- | --- | --- | --- |
| **## 根因判断** | 分析错误的本质原因，指出用户代码在哪一步逻辑、语法或认知上出现了偏差。 | 50-150字 | 导师口吻，一针见血，具体不泛泛。 |
| **## 正确思路** | 给出正确解法的核心思路，引导用户如何正确思考，而不是直接提供答案。 | 50-150字 | 启发式，强调“Why”而不是仅仅说“How”。 |
| **## 被忽略的边界** | 列出 2-3 个在当前场景下容易忽略的 edge case（边界条件）或潜在异常。 | 50-150字 | 条理清晰，具体到特定的输入或程序状态。 |
| **## 复习检查清单** | 提供 3-5 条具体的自查问题，供用户在后续复习时进行针对性的对照检验。 | 50-150字 | 祈使句为主，具有明确的行动指导意义。 |

## 3. Language 分支对照表

根据代码语言或类型，在基础 Prompt 之上附加特定的约束和重点，形成分支策略：

| Language 类型 | 检测关键词 | 额外重点 | Prompt 附加说明示例 |
| --- | --- | --- | --- |
| **算法/LeetCode** | `algorithm`, `leetcode`, `c++`, `java` | 复杂度分析、空间优化、经典算法模型匹配。 | “请额外分析当前解法的时间和空间复杂度，并指出是否有更优的算法模型（如动态规划、双指针等）。” |
| **JavaScript/TypeScript** | `javascript`, `typescript`, `js`, `ts`, `react` | 异步处理、闭包、类型安全、React 生命周期/Hook 依赖。 | “请注意指出可能存在的异步竞态条件、内存泄漏风险，或 TypeScript 类型定义的严谨性问题。” |
| **Python** | `python`, `py`, `pandas` | 语言特性（如列表推导式、生成器）、Pythonic 风格、引用传递陷阱。 | “请评估代码是否符合 Pythonic 风格，并重点指出可能由于可变默认参数或浅拷贝导致的潜在 BUG。” |
| **其他（通用）** | 未命中上述规则 | 通用的逻辑严谨性、命名规范、可读性和工程实践。 | “请重点关注代码的逻辑严密性、变量命名是否清晰以及整体结构的可读性。” |

## 4. 「从 AI 学到了什么」反馈按钮 UX 方案

为了收集 AI 分析的有效性并形成数据闭环，设计如下用户反馈机制：

- **位置**：位于 AI 分析面板的底部。在 AI 流式输出完全结束（状态变为 `completed`）后淡入显示。
- **交互**：
  - 提供两个轻量级按钮：👍「学到了」（Helpful） 和 👎「没帮助」（Unhelpful）。
  - 用户点击后，对应按钮呈现选中/高亮状态，并可短暂弹出“感谢反馈”的 Toast 提示，随后按钮不可再次点击。
- **数据存储建议**：
  - 在后端的 `review_logs`（或对应的复习记录表）中新增 `ai_feedback` 字段。
  - 字段枚举值：`'helpful'` / `'unhelpful'` / `null`。
  - *注：此项可作为 W4 的数据闭环功能点进行后端对接与持久化开发。*
- **演示价值**：在 W4 演示时，此功能能够清晰体现产品具备从“AI 赋能”到“用户评估”再到“反哺迭代”的完整产品闭环思维。

## 5. 已知风险与降级策略

在实际运行中，AI 服务的输出和网络状况可能存在不确定性，系统需做好健壮的降级处理：

- **AI 输出不含 4 个小节时前端如何处理**：
  - **风险**：AI 未严格遵循 Prompt 约束，输出格式错乱或缺失标题。
  - **策略**：前端在解析 Markdown 时，若无法通过正则或 AST 成功提取出约定的 4 个 `##` 标题段落，则**降级为按普通方式展示原始的完整 Markdown 内容**，确保分析信息对用户可见，不因解析失败而白屏。
- **网络超时时的 UX 提示**：
  - **风险**：调用大模型 API 耗时过长或网络连接中断。
  - **策略**：超过设定阈值未收到首包响应，或请求发生异常中断，停止加载骨架屏或打字机效果，并显示友好的错误状态提示（如：“AI 导师网络开小差了，请稍后重试 ☕️”），并提供明确的“重试”按钮。
- **Feature Flag Off 时的占位文案**：
  - **风险**：AI 功能通过 Feature Flag 临时关闭（如维护中），或用户配额不足。
  - **策略**：隐藏 AI 分析入口及面板，或在原位置显示占位引导文案（如：“AI 分析服务升级中，稍后回来”或“升级以解锁 AI 导师深度分析”），确保界面布局不空洞，且用户对功能状态有清晰认知。
