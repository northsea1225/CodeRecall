## 3. 原创性声明

我们以学术诚信原则郑重声明：

1. **本项目所有源代码均由队员独立编写**，未抄袭、剽窃或直接复制任何他人的项目作品代码（包括但不限于 GitHub 公开项目、博客代码片段、课程作业模板）
2. **所有第三方库与算法均通过合法开源协议引入**，并在源代码中保留原作者署名与许可证（见 §4 依赖清单）
3. **项目的产品定位、产品功能定义、UI/UX 设计、架构决策、数据模型设计均为队员原创**——本项目并非市面已有错题本工具的克隆，而是面向 OI/ACM/LeetCode 选手的特定差异化产品
4. **凡借助 AI 工具生成的代码**，均经过队员人工审阅、调整、测试后纳入仓库，详见 §5 AI 辅助开发透明声明

## 5. AI 辅助开发透明声明

### 5.1 我们的立场

活动文档原文明确鼓励：**"养成善用 AI 辅助开发的硬核习惯"**。

我们坚信，**AI 辅助开发的正确姿势不是让 AI 代写代码，而是建立一套"AI 提议 → 人决策 → AI 落地 → AI 审计 → 人验收"的闭环工作流**。本项目即是这种工作流的实践产物，我们选择主动公开 AI 辅助开发的全流程。

### 5.2 使用的 AI 工具链

| 工具 | 提供方 | 角色 |
|---|---|---|
| Claude (Opus / Sonnet) | Anthropic | 主开发助手：架构 / 代码 / 测试 / 文档 |
| Codex | OpenAI | 独立代码生成与审查（提供与 Claude 的备选方案） |
| Gemini | Google | 独立审查视角（前端 UI / 性能） |
| oh-my-claudecode (OMC) | 开源社区 | 多智能体协同框架（编排 Claude / Codex / Gemini） |
| DeepSeek API | DeepSeek | 项目运行时 AI 教练（用于 6 阶段错题分析） |

### 5.3 AI 协同的关键工作流

#### 场景 1：需求 → 架构决策

队员定义需求（如"如何实现智能复习调度"）→ AI 给出 2-3 个备选方案及对比 → 队员根据自身 OI 经验和产品定位做最终决策 → AI 实现。

**示例**：复习调度选择 SM-2 算法 vs FSRS 算法 vs Anki 改良算法 → 队员对比后选择 SM-2（理由：学术界经典，易于解释和调试） → AI 实现细节代码。

#### 场景 2：跨模型代码审计（CCG 工作流，本项目核心方法论）

项目在 2026-04-29 接受了一次完整的代码审计：

- Claude / Codex / Gemini **三方各自独立**审阅同一份代码
- 三方各自产出 issue 报告（分别 4 / 8 / 13 Medium 等级 issue，部分 issue 重叠部分独立）
- 队员合并三方报告，去重后形成 **41 个 issue** 的统一修复计划
- 4 个 Phase 分批落地，每个 Phase 完成后重新运行 245 + 49 + 13 测试验证零回归

完整审计报告见 `docs/audit/2026-04-29/`：

- `claude-report.md`（646 行）
- `codex-report.md`（227 行）
- `gemini-report.md`（149 行）
- `final-report.md`（427 行综合）
- `.claude/plan/audit-fixes.md`（1237 行执行计划）

**这是项目最具工程价值的部分**——它不是用 AI 代写代码，而是**用 AI 审 AI 自己写的代码**，建立独立审查机制。这种方法论比单独使用 AI 写代码可靠得多。

#### 场景 3：测试驱动 + AI 写测试

- 后端 245 个 pytest / 前端 49 个 vitest / 14 个 Playwright e2e
- 测试用例由队员设计意图（应该测什么、边界条件是什么），AI 实现具体测试代码
- 测试是 AI 辅助开发的"防护网"：AI 生成的代码必须通过测试才能合并

### 5.4 AI 参与度的诚实评估

| 领域 | 队员自主主导 | AI 协助实现 |
|---|---|---|
| 产品定位与差异化 | ★★★★★ 完全自主 | — |
| 6 阶段 ReviewStage 状态划分逻辑 | ★★★★★ 完全自主（基于 OI 经验） | — |
| 数据模型核心字段设计 | ★★★★ 主导 | AI 提建议 |
| 关键体验细节（暗房 / 键盘 / 一键导入） | ★★★★ 主导 | AI 实现细节 |
| CRUD 路由代码 | ★★ | ★★★★ AI 大部分 |
| React 组件骨架与样式 | ★★ | ★★★★ AI 大部分 |
| 测试用例 | ★★★ 设计意图 | ★★★ AI 实现 |
| Alembic migration SQL | ★ | ★★★★ AI 大部分 |
| 文档草稿 | ★★ | ★★★★ AI 大部分 |
| 安全决策（Cookie + CSRF / fail-fast） | ★★★★ 主导决策 | ★★★ AI 实现 + 审计 |
| CI / 自动化工作流 | ★★★ 主导 | ★★★ AI 实现 |

**总体评估**：约 30% 内容（产品定位、架构决策、关键功能设计、安全策略）由队员自主主导；约 70% 内容（代码实现、测试、文档）由 AI 协助生成但经过队员审阅 / 调整 / 测试后纳入。

### 5.5 我们坚守的边界

1. **队员理解每一行进入仓库的代码**——不存在"我也不知道 AI 写了什么"的黑盒
2. **所有 AI 输出代码经过三层测试验证**：单元 → 集成 → 端到端
3. **关键技术决策有详细文档**：见 `.claude/plan/`、`docs/audit/`、`CLAUDE.md`
4. **不复制 ChatGPT / Cursor 等通用工具的"模板代码"**：每个模块都根据本项目特定上下文调整
5. **commit message 注明 AI 协同**：本项目大量 commit 使用 `Co-Authored-By: Claude <noreply@anthropic.com>` trailer，明示 AI 协同生成
