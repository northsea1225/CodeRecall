# CodeRecall 技能执行手册

> 适用对象：当前 `CodeRecall` 三方分工方案
>
> 核心原则：`/ccg` 负责定方向，`/team` 负责真正开工，`/ask claude` 负责低频关键审阅，`/ralph` 负责单点死磕，`/ultrawork` 只在独立小任务并行时补位。

---

## 1. 一句话工作流

默认主线：

```text
/ccg -> /team -> /ask claude
```

辅助分支：

```text
卡住单点模块 -> /ralph
多个独立杂项 -> /ultrawork
最终总结/答辩优化 -> /ccg
```

不推荐作为主线：

```text
/autopilot
```

原因：当前项目已经有明确的三方分工方案，`/autopilot` 会重新接管流程，不利于保持 `Codex 主开发、Gemini 主 UX/文档、Claude 低频审阅` 这套协作结构。

---

## 2. 每个 Skill 的职责

### `/ccg`

用途：让 Codex 和 Gemini 同时给建议，然后由 Claude 汇总。

适合：
- 项目启动前对齐需求
- 方案打架时做交叉判断
- 结项前做综合复盘

不适合：
- 真正开始写代码
- 长时间作为主执行器使用

标准模板：

```text
/ccg 基于当前工作区的《三方分工方案.md》，确认本周目标、模块拆分、Codex/Gemini/Claude 的职责边界，并给出可执行任务清单。
```

---

### `/team`

用途：真正把任务拆成并行工作项并开始执行。

适合：
- W1-W4 的主要开发阶段
- 需要多人并行推进的任务
- 已经知道目标、只差开工

标准模板：

```text
/team 根据《三方分工方案.md》和刚才 /ccg 的结论，把当前阶段任务拆成并行执行项并开始干活。
```

更具体一点的版本：

```text
/team 4:executor 按《三方分工方案.md》执行 W1：1. 工程脚手架与环境打通 2. SQLAlchemy 模型与迁移 3. React 骨架与路由 4. README 1.0 与基础组件整合
```

---

### `/ask claude`

用途：低频、高价值、定点审阅。

适合：
- 架构 sanity check
- 算法正确性 review
- 导入导出/统计口径 review
- 最终 PR 级审阅

不适合：
- 日常小改反复询问
- 把 Claude 当主开发用

标准模板：

```text
/ask claude 请对 <当前阶段成果> 做一次高价值审阅，只指出关键问题。
```

---

### `/ralph`

用途：盯住单个高风险模块，直到做完并验证。

适合：
- SM-2 算法实现
- AI 错因分析接入
- 导入导出增强
- 统计聚合逻辑
- 发布前联调修复

标准模板：

```text
/ralph 完成 <单个明确模块>，直到实现、测试和边界验证都通过。
```

例如：

```text
/ralph 实现并验证 SM-2 算法与复习调度接口，确保边界 case、测试和数据写入逻辑完整通过。
```

---

### `/ultrawork`

用途：多个独立杂项高速并行推进。

适合：
- README、docs、demo-script 一起补
- 图表、空态、文案、基础组件一起补
- 多个独立 Bug 一起修

不适合：
- 主线开发阶段
- 强依赖协作和阶段验收的任务

标准模板：

```text
/ultrawork 同时完成 README 2.0、docs/architecture.md、demo-script.md、统计图表前端接入。
```

## 3. 按周使用顺序

### W1：工程地基

顺序：

```text
/ccg -> /team -> /ask claude
```

推荐命令：

```text
/ccg 基于《三方分工方案.md》，确认 W1 的目标、模块拆分和执行顺序。
```

```text
/team 4:executor 按《三方分工方案.md》执行 W1：Codex 负责脚手架、数据模型、CRUD、Monaco 最小接入；Gemini 负责 UI 规范、ASCII 原型、基础组件、README 1.0；Claude 只在最后做架构 sanity check。
```

```text
/ask claude 请对当前目录结构、API 契约、ORM 模型做一次架构 sanity check，只指出高风险问题和必要修改。
```

---

### W2：核心业务

顺序：

```text
/team -> 卡点用 /ralph -> /ask claude
```

推荐命令：

```text
/team 根据《三方分工方案.md》执行 W2：完成搜索、复习流程、SM-2、AI 错因分析前后端骨架和首轮测试。
```

```text
/ralph 实现并验证 SM-2 算法与复习调度接口，确保边界 case、测试和数据写入逻辑完整通过。
```

```text
/ask claude 请重点审阅 SM-2 算法实现、复习调度接口、AI 错因分析集成的正确性、幂等性和安全边界。
```

---

### W3：增强与重构

顺序：

```text
/team 或 /ultrawork -> 可选 /ask claude
```

推荐命令：

```text
/team 根据《三方分工方案.md》执行 W3：完成统计接口、导入导出增强、Monaco 体验增强、代码重构和测试补齐。
```

如果这一周以独立小项为主，可以改成：

```text
/ultrawork 同时完成统计图表、README 2.0、docs/architecture.md、docs/demo-script.md、前端复习引导 UI。
```

可选审阅：

```text
/ask claude 请审阅导入导出、统计聚合、版本字段和数据迁移兼容性，只指出口径不一致和潜在数据风险。
```

---

### W4：收口与答辩

顺序：

```text
/team -> /ralph -> /ask claude -> /ccg
```

推荐命令：

```text
/team 根据《三方分工方案.md》执行 W4：联调、Bug 修复、稳定性处理、打包发布、验收测试、演示材料收口。
```

```text
/ralph 修复发布前的高风险问题，重点关注数据一致性、边界 case 和回归风险，直到验证通过。
```

```text
/ask claude 请做一次最终 PR 级审阅，关注跨文件一致性、回归风险、命名、演示前盲点。
```

```text
/ccg 基于当前项目成品，从工程完整性、用户体验、演示表达三个角度做最终评估，并给出答辩前最后修改清单。
```

---

## 4. 最推荐的日常操作模式

如果你不想每次思考“该用哪个 skill”，就按下面执行：

### 开一周工作

```text
/ccg 先定方向
```

### 真正开始开发

```text
/team 负责主线执行
```

### 某一块太关键、太容易出错

```text
/ralph 负责单点死磕
```

### 需要 Claude 省额度地看关键问题

```text
/ask claude 做低频审阅
```

### 收尾和答辩

```text
/ccg 做最终综合建议
```

---

## 5. 一句话结论

> 对这份 `CodeRecall` 三方分工方案，最佳主线不是 `/ccg -> /ccg -> /ccg`，而是：
>
> **`/ccg` 定方向 → `/team` 干主线 → `/ask claude` 做低频关键审阅 → `/ralph` 死磕难点 → `/ccg` 结项复盘。**
